Technical Implementation

How Blade Archive Meets The Brief

The site is built as a readable student-level project while still demonstrating polished interaction and real-time rendering.

Bootstrap 5

Responsive grid layout, mobile navbar, accessible buttons, and consistent spacing across desktop, tablet, and phone screens.

Three.js + WebGL

Real-time scene, PerspectiveCamera, WebGLRenderer, ACES tone mapping, cinematic grading, fog, shadows, bloom, and selectable lighting rigs.

GLTFLoader

Loads Blender-exported GLB files dynamically, centers them with bounding boxes, scales them consistently, and preserves the model workflow.

HDR Reflections

RGBELoader and PMREMGenerator create environment reflections so metallic blades respond to the lighting like polished game assets.

OrbitControls

Smooth damping allows the user to inspect each sword, with camera buttons for full, blade, hilt, and profile viewpoints.

Audio System

Ambient audio, mute control, hover cues, and unsheath effects only start after user interaction, with procedural fallbacks.

Optimisation

Renderer pixel ratio is capped, particles are lightweight, previews pause when off-screen, and resize handling updates only the required buffers.

About The Project

Built Like A Premium Weapon Showcase

Blade Archive presents six original Blender-modelled weapons in a dark fantasy interface inspired by modern game item-viewer screens. Every technical choice — from the three lighting rigs to the four camera presets and the custom GLSL post-processing pipeline — was made to demonstrate clearly assessable evidence while remaining readable as student-level code.

JavaScript is structured as a loose MVC separation: a SWORDS data array acts as the model layer, the Three.js renderer and EffectComposer form the view layer, and the initUI event-binding block acts as the controller. Audio is isolated inside a BladeAudio class so it can be tested independently, and the animation loop is kept as a single animate() function that ticks both the viewer and the lightweight hero preview.

01 Design Decisions

Visual Language

The charcoal/silver/ember palette was chosen to match the aesthetic of game weapon-viewer screens (e.g. Elden Ring, Dark Souls item screens) where dark backgrounds push the subject forward. Georgia serif is used for headings to suggest craftsmanship; Inter handles all body text for screen legibility.

Layout Architecture

Bootstrap 5's responsive grid manages all breakpoints. The viewer itself uses CSS Grid with a fixed-width sidebar and a fluid canvas column so the Three.js canvas always fills the remaining viewport without JavaScript measuring. scroll-padding-top on the html element keeps nav anchors clear of the sticky header on all screen sizes.

Lighting & Camera

Three lighting rigs serve different purposes: Material shows PBR accuracy under neutral keys; Forge adds an orange key light to simulate a smithy environment; Studio uses pure-white keys and high env intensity for clean technical inspection. The four camera presets (Full / Blade / Hilt / Profile) let any viewer verify geometry at the right scale without needing to orbit manually.

Code Structure

All scene data lives in the SWORDS and LIGHT_RIGS constants at the top of main.js (model layer). The Three.js renderer, EffectComposer, and shader passes form the view. The initUI function wires all DOM events to renderer state (controller). This separation means lighting, camera, and model data can be changed without touching rendering code.

02 3D Model Production

All six weapons were modelled from scratch in Blender 4.x. The workflow prioritised clean, efficient geometry: models use loop-cut subdivision only where edge detail is needed, mirror modifiers handle bilateral symmetry, and hard-surface bevels on blade edges prevent specular pinching under point lights. Materials were set in Blender using the Principled BSDF shader with metallic = 1.0 and roughness tuned per weapon (a rapier is near-mirror at 0.08; a halberd is battle-worn at 0.38). All models were exported as GLB with embedded materials but no baked textures — the PBR material response under the HDR environment achieves the final look entirely in Three.js.

After loading, Three.js centers each model using its bounding box and scales it to a per-sword targetSize constant, so every weapon fills the viewport consistently regardless of the raw Blender units. The Blender timelapse in the Making Of section shows the full modelling and material workflow for the Lance.

Blender Workflow

  1. Blockout silhouette with cylinders and cubes, checking proportions from the front and side orthographic views
  2. Add loop cuts for blade taper, crossguard shape, and grip wrapping — no tri-fans, quads throughout
  3. Apply mirror modifier along the X axis for blade geometry; remove before export to keep GLB clean
  4. Bevel hard edges (1–2 segments, 0.02–0.06 px) to catch specular highlights without adding dense geometry
  5. Assign Principled BSDF materials: metallic 1.0, roughness per weapon, IOR 1.45 for pommel gems
  6. Export as GLB (Geometry + Materials, no animations, Y-up) and verify in the Three.js viewer

03 Accessibility & Testing

Accessibility

  • All interactive controls carry aria-label and aria-pressed attributes; segmented button groups use role="group" with a group label
  • prefers-reduced-motion: reduce is detected on load — auto-rotation is disabled and all CSS transitions are suppressed to one frame
  • Audio is muted by default and only activates on explicit user gesture, respecting browser autoplay policy and user preference
  • Primary text colour (#eef2f3) on the darkest background (#050607) gives a contrast ratio above 15:1, well above the WCAG AA 4.5:1 requirement
  • Canvas elements use aria-label to describe their subject to screen readers; decorative elements carry aria-hidden="true"
  • The Bootstrap navbar collapses correctly on mobile and manages aria-expanded state on the toggler button
  • Semantic landmark elements are used throughout: <nav>, <main>, <aside>, <article>, <section>, <footer>
